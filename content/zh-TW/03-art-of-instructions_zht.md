# 第 3 章：指令的藝術

---

## TL;DR

> 你說什麼，比 Claude Code 有多聰明更重要。
> 爛問題 + 聰明的 AI = 爛答案。
> 好問題 + 普通的 AI = 驚豔的答案。
> 這章是整本書 CP 值最高的一章。

---

## 指令的本質：你在做什麼？

跟 Claude Code 對話，你其實在同時做三件事：

1. **傳達目標**：你要什麼結果
2. **傳達背景**：它需要知道什麼前提
3. **傳達限制**：什麼不能做、什麼一定要做

大多數人只做了第一件事。

---

## 對比：同一個任務，兩種問法

### 爛問法

```
> 幫我優化這個函數的效能
```

Claude 心裡在想：
- 哪個函數？
- 什麼效能問題？記憶體？速度？耗電？
- 優化到什麼程度？
- 有什麼限制不能改？

結果：給你一個通用答案，很可能完全不適用。

---

### 好問法

```
> 看一下 UserProfileViewController.swift 裡的 loadUserData() 函數。
> 這個函數每次 viewDidAppear 都會被呼叫，在滑動時造成明顯卡頓。
>
> 我想要：
> - 加入快取，讓同一個 userID 10 分鐘內不重複請求
> - 不改函數簽名（其他地方有在呼叫）
> - 保留現有的錯誤處理邏輯
>
> 先讀這個檔案了解目前的實作，然後告訴我你的方案。
> 確認之後再動程式碼。
```

Claude 心裡在想：清楚。

結果：精準的解法，可以直接用。

## 黃金公式：目標 + 背景 + 限制 + 動作要求

好的指令包含這四個元素（不是每次都要全寫，但要想清楚）：

```
[目標] 我要 X
[背景] 背景是 Y，相關檔案是 Z
[限制] 不能改 A，一定要保留 B
[動作] 先做什麼，再等我確認 / 直接修改 / 給我建議
```

## 六個核心指令模式

### 模式一：探索與理解

適合：理解陌生的程式碼、分析架構

```
> 幫我理解 [檔名/模組名]。
> 我特別想知道：[具體問題]
> 用白話解釋，假設我對這個模組的歷史不熟
```

範例：
```
> 幫我理解 NetworkManager.swift。
> 我特別想知道：重試邏輯是怎麼實作的，以及錯誤處理的策略。
> 用白話說，假設我剛加入這個專案。
```

---

### 模式二：修 Bug

適合：修復 bug、解決 crash

```
> 我遇到這個錯誤：[完整錯誤訊息]
> 發生的情境：[具體場景]
> 相關檔案是：[檔名]
> 我已經試過：[試過什麼]，沒有效果
> 幫我找出原因並修復
```

**關鍵原則：貼完整錯誤訊息，不要只說「有錯誤」。**

---

### 模式三：實作功能

適合：新增功能

```
> 我需要實作 [功能描述]
> 現有的相關程式碼在 [檔名]
> 要符合專案的 [架構模式/程式碼風格]
> 先給我實作方案，確認後再寫程式碼
```

**養成習慣：先看方案再寫 code。** 避免做到一半才發現方向錯了。

---

### 模式四：重構與改善

適合：提升程式碼品質

```
> [檔名] 裡的 [函數/類別] 需要重構
> 主要問題：[問題描述]
> 重構目標：[更好維護 / 更好效能 / 更 idiomatic Swift / 等等]
> 限制：外部介面不能改（或：可以改）
> 先分析現在這段程式碼的問題，再給我重構方案
```

---

### 模式五：產生樣板

適合：快速產生重複性的程式碼

```
> 按照 [現有範例檔] 的模式，幫我為 [新功能] 產生樣板
> 遵循現有程式碼的命名規範
> 需要包含：[具體內容]
```

**一定要給現有範例。** 讓 Claude 模仿你的風格，而不是自己發明一套。

---

### 模式六：分析與 Review

適合：code review、品質檢查

```
> 幫我 review [檔名] 的程式碼，重點關注：
> 1. 潛在的記憶體洩漏
> 2. 執行緒安全問題
> 3. 邊界條件處理
>
> 以列表形式給出問題，按嚴重程度排序
> 不要修改程式碼，只給建議
```

## 指令的魔法詞彙

這些詞彙可以顯著改善 Claude 的行為：

| 詞彙 | 效果 |
|------|------|
| 「先…再…」 | 控制執行順序，避免 Claude 跳過步驟 |
| 「確認後再修改」 | 避免 Claude 直接動手改程式碼 |
| 「只改 X，不要動 Y」 | 精準控制修改範圍 |
| 「給我 3 個選項」 | 獲得多個選擇，不被第一個答案鎖定 |
| 「假設我是…」 | 控制解釋的深度 |
| 「用[語言]回答」 | 控制輸出語言 |
| 「簡潔，不需要解釋」 | 讓它直接給程式碼，省掉廢話 |

## Context 的藝術：多少才剛好

**太少**：Claude 猜你的意圖，答案跑偏

**太多**：Context 被不相關的資訊塞滿，核心問題反而淹沒

**剛好**：

```
精準 > 大量
```

不要說「我整個專案都是這樣設計的… (1000 字)」
要說「關鍵是這個 class 繼承的是 UIViewController 而不是 UITableViewController，
所以 dequeueReusableCell 沒辦法用」

## 多輪對話策略

Claude Code 的多輪對話是有狀態的，好好利用這一點。

**錯誤做法：每次都從頭開始**

```
第一輪：幫我看 UserService.swift 有什麼問題
第二輪：幫我再看一次 UserService.swift，我之前說效能不好，
        現在想用快取...
```

**正確做法：在前面的基礎上遞進**

```
第一輪：讀 UserService.swift，告訴我你對這個 class 的理解
第二輪：好，那 fetchUser 這個方法的效能怎麼樣？
第三輪：我想加快取，你有哪些方案？
第四輪：用第二個方案，但快取時間改成 5 分鐘
```

在這個對話裡，Claude 的 context **是累積的**，每一輪都建立在前面的理解上。

---

## 當 Claude 給了錯誤答案

**不要這樣：**
```
> 你錯了，再試一次
```

**要這樣：**
```
> 你上一個解法有問題：[具體說明哪裡有問題]
> 正確的理解應該是：[你的理解]
> 根據這個，給我新的方案
```

**糾正 Claude 跟糾正人類同事一樣：說清楚問題在哪裡，提供正確的 context，然後請它重試。**

## 實戰：iOS 開發者提示模板

```bash
# 分析 Crash
> 這是 crash log：[貼上 log]
> App 版本 iOS 17，裝置 iPhone 15
> 幫我找出 crash 原因並修復

# SwiftUI Preview 問題
> SwiftUI Preview 出現這個錯誤：[貼上錯誤]
> 相關檔案是 [檔名]
> 修復的同時不要改到實際的 View 邏輯

# 記憶體洩漏
> Instruments 在 [class 名稱] 發現記憶體洩漏
> 相關程式碼在 [檔名]
> 找出洩漏位置，重點關注 closure 裡的 self 引用

# iOS 版本相容性
> [功能] 需要支援 iOS 15-17
> 目前程式碼只考慮了 iOS 17
> 加入版本判斷和向下相容的實作
```

## 本章小結：5 個原則

1. **目標 + 背景 + 限制 + 動作**：好指令的四個元素
2. **精準 > 大量**：不是越多越好，是越準確越好
3. **先方案後程式碼**：複雜任務先確認方向再動手
4. **遞進式多輪**：善用對話的連續性
5. **具體糾正**：說錯了，要說清楚哪裡錯，不是說「不對」
