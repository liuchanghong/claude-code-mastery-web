# 第 12 章：常見錯誤與修正

---

## TL;DR

> 大多數人都會踩相同的坑。
> 這章把血淚教訓提煉出來，讓你不用浪費幾個月摸索。
> 每個錯誤都有具體的修正方法。

---

## 錯誤一：把 Claude Code 當程式碼生成機器

### 症狀

```
「幫我寫一個使用者登入功能」
→ Claude 生成了一些程式碼
→ 直接複製貼上進專案
→ 完全不符合現有的程式碼風格和架構
→ 大量返工
→ 不如自己寫更快
```

### 為什麼發生

Claude Code 不了解你的專案架構。生成的程式碼是通用的，不是針對你的。

### 修正

```
# 給它 context，讓它匹配你的模式
> 讀 Sources/Features/Profile/ 裡的程式碼
> 我想在 Cart 模組加入類似的登入狀態檢查邏輯
> 按 Profile 模組的模式來，不要自己發明新的
```

**原則：讓 Claude 模仿，而不是發明。**

## 錯誤二：在錯誤的時機要求程式碼

### 症狀

```
發現問題 → 立刻叫 Claude 修程式碼 → 出現更多問題 → revert → 再叫 Claude → 無限循環
```

### 為什麼發生

沒有先搞清楚問題就動手。治標不治本。

### 修正

```
鐵則：任何改動之前，一定先讓 Claude 說清楚根本原因

> 分析這個問題的根本原因
> 告訴我可能的解法和各自的取捨
> 我選定方案後，再寫程式碼
```

**「先診斷，再開藥」——這個原則對除錯和功能開發都適用。**

## 錯誤三：用自然語言描述程式碼

### 症狀

```
「我有一個 Manager class，裡面有一些處理使用者資料的方法，我想重構成...」
→ Claude 根據你的描述生成程式碼
→ 和實際程式碼差很多
→ 直接給它看程式碼就好了
```

### 修正

**不要描述程式碼——給它看程式碼。**

```bash
# 不好
> 我有一個 UserManager，有 fetchUser、updateUser、deleteUser 方法...

# 好
> 讀 Sources/Core/UserManager.swift
> 我想把它重構成 Repository 模式
```

## 錯誤四：單一 Session 用太長

### 症狀

- 同一個 Claude session 用了 2 小時以上
- Claude 開始忘記 session 初期做的決定
- 同樣的問題得到不同的答案
- 回答越來越模糊

### 為什麼發生

Context window 滿了，前面的資訊被壓縮或遺忘。

### 修正

```bash
# 每 30-40 分鐘，儲存目前的狀態
> 把我們 session 裡的關鍵決定和結論存到 CONTEXT.md

# 開新 session
> 讀 CONTEXT.md，繼續我們之前的工作
> 現在需要做的是 [下一步]
```

## 錯誤五：盲目相信 Claude 的答案

### 症狀

```
Claude 說「問題是 X」→ 你修 X → 問題沒解決
Claude 說「用這個 API」→ 你用了 → 這個 API 在 iOS 17 已經 deprecated
Claude 給了程式碼 → 你直接用 → 有個細微的 bug
```

### 為什麼發生

Claude 會出錯。特別是在這些情況下：
- 輸入模糊
- 版本相關的問題
- 需要你個人判斷的決策
- 複雜的多檔案依賴關係

### 修正

**對 Claude 的所有輸出保持批判性思考。**

建立「驗證後再行動」的習慣：

```bash
# 驗證新 API
> 你提到的這個 API 在 iOS 16 可以用嗎？最低版本是多少？

# 驗證邏輯
> 用這個測試案例走一遍你的邏輯：[具體案例]
> 結果應該是 [預期結果]——你的方案能正確處理嗎？

# 驗證影響
> 這個改動會影響 [相關功能] 嗎？為什麼？
```

## 錯誤六：把「省時間」當唯一目標

### 症狀

```
讓 Claude 寫所有程式碼
→ 你不理解這些程式碼
→ 出問題時不知道怎麼 debug
→ 後續修改破壞了之前的邏輯
→ 技術債累積
```

### 修正

**Claude Code 應該幫你更快地做你理解的事——而不是幫你跳過理解。**

當 Claude 生成複雜程式碼時：

```bash
> 在你寫程式碼之前，先解釋你的實作思路
> 我需要確認我理解了方案再繼續

> 寫完後，解釋這段程式碼裡最關鍵的 3 個地方
> 我之後需要能夠維護這段程式碼
```

## 錯誤七：不維護 CLAUDE.md

### 症狀

- 每次都要重新解釋專案背景
- Claude 的程式碼風格和專案不一致
- 一直在糾正相同的問題

### 修正

養成維護 CLAUDE.md 的習慣：

```bash
# 發現新的專案規範時
> 把這個規範加到 CLAUDE.md：[描述規範]

# 發現錯誤的規範時
> 更新 CLAUDE.md，把 X 的規範改成 Y

# 每週五，花 5 分鐘
> 回顧這週我對 Claude 做的修正
> 看哪些需要寫進 CLAUDE.md
```

## 錯誤八：一次給 Claude 太複雜的任務

### 症狀

```
> 幫我把整個 App 從 UIKit 遷移到 SwiftUI，
> 順便把架構從 MVC 改成 MVVM，
> 再加上單元測試，順便優化效能...
→ Claude 開始做，做到一半 context 滿了
→ 改動亂成一團，無法 revert
→ 專案卡在半改動的狀態
```

### 修正

**一次只做一件事。**

任何看起來「複雜」的任務——先拆解：

```bash
> 我想做 [大任務]
> 幫我拆解成：
> - 每個步驟可以獨立 commit
> - 每步完成後專案還能執行
> - 按依賴關係排序
>
> 列出所有步驟，再從第一步開始
```

## 錯誤九：忽略 Claude 表達的不確定性

### 症狀

```
Claude：「這可能是問題 X，但我不確定，需要更多資訊確認...」
你：直接去修 X
→ 修錯了
```

### 修正

當 Claude 表達不確定時，**先消除不確定性再行動**：

```bash
> 你說「可能」是問題 X。幫我設計一個驗證方法。
> 如果真的是 X，應該會有什麼可觀察到的症狀？
> 如果不是 X，看起來會有什麼不同？
```

## 錯誤十：用 Claude Code 查文件

### 症狀

```
> Swift 的 @MainActor 是什麼意思？
→ 得到一個解釋
→ 官方文件更準確
```

### 修正

純粹的 Swift/iOS API 查詢，直接看 [Apple 官方文件](https://developer.apple.com) 或 Dash——更準確。

Claude Code 的優勢是在**你的具體程式碼脈絡**裡分析和應用：

```bash
# 不值得用 Claude 做的事
> Swift 的 @MainActor 是什麼？

# 更有價值的用法
> 這段程式碼（貼上程式碼）應該用 @MainActor 還是
> DispatchQueue.main.async？分析我的具體情況給我建議
```

## 快速參考：常見問題診斷

| 問題 | 最可能的原因 | 修正 |
|-----|------------|-----|
| Claude 生成的程式碼和專案風格不符 | 沒給範例 / 沒有 CLAUDE.md | 給類似的現有程式碼作為參考 |
| Claude 的分析不準確 | Context 不足 | 提供更多相關檔案 |
| Claude 一直給同樣的錯誤答案 | 問法有誤導性 | 換個角度問 |
| Session 中途品質下降 | Context window 滿了 | /compact 或開新 session |
| Claude 遺漏了關鍵依賴 | 沒說明檔案關係 | 明確解釋檔案之間的連接 |
| Claude 給了有 bug 的程式碼 | 邊界條件沒說清楚 | 在需求裡明確列出邊界條件 |
