# 第 13 章：專家思維——從「會用」到「精通」的最後一躍

---

技術技巧你已經學完了。但成為專家還需要更多：一種讓你在任何情況下都能有效使用 Claude Code 的思維方式。這章是哲學——而哲學決定了你的天花板。

---

## 專家和普通使用者的真正差別

不是技巧數量，而是**問題的品質**。

普通使用者問：**「這段程式碼怎麼修？」**

專家問：**「幫我理解這段程式碼在整個系統裡扮演的角色，
然後分析改動它可能觸發的連鎖反應，
最後基於這個分析給我一個修法。」**

差別不在於問題的長度，而在於**思考的深度**。

---

## 核心思維一：系統性思考

**把程式碼看成一個系統，而不是孤立的檔案。**

每次遇到問題，先問：

```
這個問題在哪一層？
  - 資料層（儲存、網路、快取）
  - 業務邏輯層（ViewModel、Service）
  - 呈現層（View、ViewController）

改動這一層會影響哪些其他層？
上下游的依賴是什麼？
```

然後用這個框架告訴 Claude：

```bash
> 這個 bug 在業務邏輯層（CartViewModel），
> 但根本原因可能在資料層（CartRepository 的快取策略）
> 讀這兩個檔案，從系統角度分析這個問題
```

---

## 核心思維二：分層驗證

**大任務要分層驗證——不要等到最後才發現走錯方向。**

```
              /\
             /  \
            /  UI  \          ← 最後驗證（成本最高）
           /        \
          / 業務邏輯  \       ← 中途驗證
         /            \
        /               \
       /  資料/演算法  \   ← 最先驗證（成本最低）
      /                   \
     /   核心假設         \ ← 開始之前就驗證
    /_______________________\
```

每次開始一個任務前，先問：

```bash
> 我的假設是 [X]。在開始實作之前，
> 幫我找出這個假設有沒有明顯的缺陷
```

---

## 核心思維三：把 Claude 當「橡皮鴨」+「專家顧問」

**橡皮鴨除錯法的升級版：**

面對複雜問題時，不要立刻叫 Claude 給答案。而是**把問題描述給 Claude**：

```bash
> 我要描述一個我正在思考的問題——你不需要給我解法。
> 只要問我問題，幫我釐清我的思路就好。
>
> 問題：我的 App 在某些裝置上啟動很慢，但我不確定瓶頸在哪裡
> 開始問我問題
```

Claude 會扮演蘇格拉底式的導師，透過提問幫你自己找到答案。這個方法讓你對問題的理解程度，遠超過直接問答案。

---

## 核心思維四：情境化學習

**用你自己專案的真實程式碼來學新技術，而不是抽象的教學。**

想學 Swift Concurrency？

```bash
# 不要這樣做
> 解釋 Swift Concurrency 裡的 actor 是什麼

# 這樣做
> 讀我的 DataStore.swift
> 裡面有很多 DispatchQueue.main.async
> 如果用 @MainActor 改寫，會是什麼樣子？
> 給我一個對比——用我自己的程式碼解釋 actor 的概念
```

用自己的程式碼學習，理解深度是看教學的 3 倍。

---

## 核心思維五：建立回饋循環

**每次 Claude 表現好或不好，都是一次改善的機會。**

```
Claude 表現不好
    ↓
分析：context 不足？問法有問題？任務太複雜？
    ↓
調整：更精準的措辭 / 更多 context / 任務分解
    ↓
測試：對同類問題再試一次
    ↓
沉澱：把有效的做法記錄到 CLAUDE.md 或個人筆記
```

幾個月的累積之後，你會有一套屬於自己、高度有效的 Claude Code 使用方法。

## 專家的 10 個日常習慣

### 習慣一：改之前先讀

任何改動前，讓 Claude 讀相關程式碼並總結它的理解，再討論修改方案。

### 習慣二：先提假設再要答案

複雜問題先提出假設，讓 Claude 幫你驗證，而不是直接問答案。

### 習慣三：測試先行

任何新功能，先讓 Claude 生成測試描述（不是程式碼），確認所有情境都涵蓋了，再實作。

### 習慣四：定期維護 CLAUDE.md

每週花 10 分鐘，分析這週你對 Claude 做的修正。值得保留的寫進 CLAUDE.md。

### 習慣五：複雜任務分而治之

任何實作超過 1 小時的任務——先拆成每個不超過 30 分鐘的子任務，獨立完成和驗證。

### 習慣六：保持對話精簡

長 session 裡，定期用 `/compact` 或整理重點開新 session。不要讓 context 無限膨脹。

### 習慣七：給 Claude 具體的負面回饋

Claude 答得不好時，告訴它哪裡有問題，而不是直接重來。

### 習慣八：向 Claude 學習

Claude 提出你不熟悉的方案時，先理解它，再決定接受還是拒絕。這是學習的機會。

### 習慣九：記錄成功的提示詞

找到特別有效的措辭？記下來，放進你的提示詞庫。

### 習慣十：知道什麼時候不用 Claude Code

簡單的單一檔案修改、你完全理解的問題、快速的 API 查詢——這些不需要 Claude Code。把它用在真正值得的地方。

## 一個誠實的自我評估

你現在是哪個層次的 Claude Code 使用者？

**初學者（會用）：**
- 能叫 Claude 解釋程式碼
- 能叫 Claude 修簡單的 bug
- 能叫 Claude 產生樣板

**中階（用得好）：**
- 用 CLAUDE.md 配置專案背景
- 多輪對話中管理 context
- 把複雜任務拆解成步驟
- 批判性地 review Claude 的輸出

**進階（熟練）：**
- 工作流裡有自訂指令和 Hooks
- 能處理複雜的多檔案、跨模組任務
- Claude Code 融入了日常工作節奏
- 能帶團隊成員使用 Claude Code

**專家：**
- 用 MCP 連接外部系統
- 把 Claude Code 整合進 CI/CD
- 一天之內能對陌生專案建立完整理解
- 把 Claude Code 當思考工具，而不只是實作工具

## 最後一件事：保留人類的判斷

**每個工具都是工具。Claude Code 也不例外。**

它會出錯。它的知識有截止日期。它不了解你的業務。

身為開發者，你的責任是：
- **你理解你發佈的每一行程式碼**（即使是 Claude 生成的）
- **你能解釋每一個架構決策**
- **你對程式碼品質負責**——而不是「AI 叫我這樣做的」

Claude Code 是讓你更強大的工具，不是取代你思考的東西。

**好好用它——但永遠保留你的判斷。**
