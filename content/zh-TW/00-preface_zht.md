# 前言：打破你對 AI 助理的幻想

---

大多數人用 Claude Code 效果不好，原因不在模型，在於 context（上下文）。

---

## 從一個真實的場景說起

凌晨一點，你盯著一段 Swift code。

Crash 在 `viewDidLoad` 裡，但怎麼看都看不出問題。你打開 Claude，把 code 貼進去，說：「幫我看看這段有什麼問題。」

Claude 給了一段分析，聽起來很有道理。你照著改，跑起來，還是 crash。

你再問，它再回答，你再改。

一小時後，你才發現：問題是三個檔案之外的一個 singleton，在某個 race condition 下造成的記憶體問題。

**Claude 沒騙你。它就是不知道那三個檔案的存在。**

這就是大多數人用 Claude Code 的方式：把一個盲人帶進暗室，叫他幫你找電燈開關。

---

## 問題不在 Claude 不夠聰明

問題是 **context（上下文）**。

Claude Code 只知道你給它的東西。你給一個 function，它就只看那個 function。你給整個 repo，它才能真正幫你分析。

這本書最重要的第一件事，就是讓你真正搞懂「context」這件事。掌握 context，就掌握了 Claude Code 八成的威力。

---

## 用 Claude Code 的人，大概分三種

**第一種：複製貼上型**

把 code 貼進去，把答案貼出來。

效果：效率提升 20%，但總覺得差點什麼。

**第二種：對話驅動型**

把 Claude Code 當聰明同事，用對話解決問題。

效果：效率提升三到五倍，但需要很多引導。

**第三種：系統整合型**

把 Claude Code 融入整個工作流，讓它在對的時機做對的事。

效果：不只是效率——整個開發品質都會跳一個台階。

這本書要把你從第一種，訓練成第三種。

---

## 讀完你會發現

1. Claude Code 最強的地方不是寫 code，而是幫你搞懂一個陌生的 codebase
2. 你的提示詞品質，比模型本身更重要
3. CLAUDE.md 是被嚴重低估的功能，沒有之一
4. 工作流整合才是真正的乘數
5. 真正會用 Claude Code 的 iOS 工程師，一個人可以做三個人的事
