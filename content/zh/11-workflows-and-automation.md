# 第 11 章：工作流与自动化

---

## TL;DR

> 单次使用 Claude Code 能提升效率 2-3 倍。
> 把它系统地融入日常工作流，能提升 5-10 倍。
> 这章展示真实的、可直接复制的工作流。

## 一个 10x 开发者的典型工作日

### 早上 9:00：开始工作

不要先看邮件，先做这个：

```bash
cd your-project
claude
> 快速状态检查：
> 1. git status —— 我昨天留下了什么未完成的工作？
> 2. git log --since="yesterday" --oneline —— 昨天做了什么？
> 3. 读 PROGRESS.md（如果有）—— 有没有待办事项？
>
> 给我一个 3 分钟的"今天从哪里继续"的摘要
```

### 9:10：进入状态

```bash
> 好，今天继续做 [功能名]
> 我们昨天停在 [上下文]
> 现在要做 [下一步]
> 开始吧
```

### 中午：快速 check-in

```bash
> 快速看看我今天的代码改动质量（git diff HEAD~3..HEAD）
> 有没有需要立即关注的问题？
```

### 下班前：收尾

```bash
> 今天要结束了，帮我：
> 1. 把未完成的工作清单写入 PROGRESS.md
> 2. 写今天所有改动的 commit message
> 3. 快速 review 所有 staged 改动，有没有 debug 代码遗留
```

## 工作流 1：新功能开发流程

```
1. 需求确认（5分钟）
   > 我要做 [功能]，用简单语言描述实现思路，让我确认理解是否正确

2. 技术调研（10分钟）
   > 读相关代码，告诉我现有基础设施和需要新建的东西

3. 方案设计（15分钟）
   > 给我实现方案，分解成独立可测试的步骤

4. 逐步实现（N个循环）
   > 实现步骤 1 → review → 确认 → 下一步

5. 测试（20分钟）
   > 生成单元测试 → 运行测试 → 修复问题

6. 收尾（10分钟）
   > pre-commit review → 写 commit message → 提交
```

---

## 工作流 2：Bug 修复流程

```
1. 信息收集（5分钟）
   > 收集：crash log + 复现步骤 + 相关代码 + 已尝试方法

2. 根因分析（10分钟）
   > 给 Claude 所有信息，让它分析根因（先分析，不改代码）

3. 假设验证（10分钟）
   > 如果需要，写诊断代码帮助确认假设

4. 修复（按复杂度）
   > 确认方案 → 实现修复 → 写回归测试

5. 验证（10分钟）
   > 在相同场景下验证修复 → 测试相关功能是否受影响
```

## 工作流 3：Code Review 流程

### 作为提交者

```bash
# 提交前的自我 review
claude
> 运行 git diff --staged
> 以严格的 reviewer 身份审查这些改动
> 用苹果工程师的标准
> 不要给鼓励，只给问题

# 完善后提交
claude
> 写 commit message，遵循 Conventional Commits
> 简洁，重点说"为什么"改，不说"改了什么"
```

### 作为 Reviewer

```bash
# 快速了解 PR
claude
> 这个 PR 改动如下：（粘贴 diff 或文件列表）
> 我只有 15 分钟做 review
> 帮我快速识别最重要的 3-5 个问题
> 排除样式问题，专注功能和安全

# 深度 review
claude
> 对这个 PR 做完整 review，特别关注：
> - 状态管理的正确性
> - 线程安全
> - 错误处理是否完整
> - 测试覆盖率
```

## 工作流 4：代码库理解流程（接手老项目）

这是 Claude Code 最被低估的场景之一。

### 第 1 天：整体理解

```bash
claude
> 我刚接手这个项目，帮我建立整体理解。
> 按顺序做：
> 1. 读 README 和主要文档（如果有）
> 2. 探索目录结构
> 3. 找入口文件和核心模块
> 4. 给我一个"新员工第一天需要知道的5件事"
```

### 第 2 天：深入核心

```bash
claude
> 今天我要深入理解业务核心：支付流程
> 追踪从用户点击"立即购买"到支付完成的完整代码路径
> 画出流程图（文字版），标出每个关键决策点
```

### 第 3 天：找出风险

```bash
claude
> 基于你对这个项目的了解，告诉我：
> 1. 最脆弱的代码在哪里（改一行可能炸的那种）
> 2. 最缺测试的关键路径是什么
> 3. 技术债最重的模块是什么
> 4. 有什么我应该在接手早期就知道的"坑"
```

## 工作流 5：重构流程

```
策略：不要一次性重构大模块。用"绞杀者模式"：

1. 新建正确的实现（平行于旧实现）
2. 把调用者逐步迁移到新实现
3. 旧实现变成空壳后删除

指令示例：
> 我想重构 NetworkManager，但不能影响当前运行
> 使用绞杀者模式：
> 1. 创建新的 NetworkClient（实现相同功能但用 async/await）
> 2. 让旧的 NetworkManager 内部调用 NetworkClient
> 3. 列出所有直接用 NetworkManager 的地方（给我迁移计划）
> 4. 我们分批迁移
```

## 工作流 6：文档生成流程

```bash
# 为一个模块生成文档
claude
> 为 Sources/Core/Network/ 生成 README.md：
> - 模块的职责和设计决策
> - 主要类和接口的说明
> - 使用示例（真实的代码示例）
> - 常见问题 / 注意事项

# 为 API 生成文档注释
claude
> 为 APIClient.swift 的所有 public 接口加 DocComment
> 风格：苹果官方文档风格
> 内容：参数、返回值、throws 的条件、使用示例（至少一个）

# 更新 CHANGELOG
claude
> 根据 git log v2.4.0..HEAD
> 更新 CHANGELOG.md
> 格式：Keep a Changelog 规范
> 用非技术语言描述每个改动
```

## 工作流 7：性能优化流程

```
1. 测量（先测量，再优化）
   > 帮我设计一个基准测试，测量 [功能] 的当前性能

2. 分析
   > 这是 Instruments 的数据：[粘贴数据]
   > 找出最大的性能瓶颈

3. 优化
   > 只优化最大的瓶颈（不要过早优化）
   > 给我方案，我确认后实施

4. 验证
   > 用同样的基准测试验证优化效果
   > 确保没有功能回归
```

## 把工作流变成团队标准

如果你的团队也在用 Claude Code，可以把工作流标准化：

### 团队 CLAUDE.md 模板

```markdown
# 团队工作流规范

## Code Review 标准
使用 `/review` 命令做 pre-commit review，所有 P0 问题必须修复后才能提交。

## Commit Message 规范
使用 Conventional Commits，用 `/standup` 命令帮助生成。

## Bug 修复规范
所有 bug 修复必须附带回归测试。

## 新功能开发规范
功能开发必须先经过技术方案设计阶段，方案记录在 PR description 里。
```

## 自动化脚本：一键式工作流

把常用的 Claude Code 操作包装成 Shell 脚本：

```bash
# ~/bin/cc-morning —— 早上的状态检查
#!/bin/bash
cd $1 || exit 1
claude -p "
检查项目状态：
1. git status
2. git log --since=yesterday --oneline
3. 如果有 PROGRESS.md，读取内容

给我今天应该从哪里继续的摘要（3句话内）
"
```

```bash
# ~/bin/cc-review —— 提交前 review
#!/bin/bash
claude -p "
审查 git diff --staged 里的所有改动
如果有 P0 问题，输出 BLOCKED: [问题描述]
否则输出 OK
只输出这一行，不要其他内容
"
```
