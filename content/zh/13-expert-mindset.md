## 专家和普通用户的真正区别

不是技巧的数量，是**提问的质量**。

普通用户问：**"这段代码怎么修？"**

专家问：**"帮我理解这段代码在整个系统里扮演的角色，
然后分析改动这里可能带来的连锁反应，
最后基于这个分析给我修复方案。"**

区别不在于问题的长度，而在于**思维的深度**。

---

## 核心思维模式 1：系统思维

**把代码看成系统，不是孤立的文件。**

每次遇到问题，先问：

```
这个问题在哪一层？
  - 数据层（存储、网络、缓存）
  - 业务逻辑层（ViewModel、Service）
  - 表现层（View、ViewController）

改动这一层会影响哪些层？
上下游的依赖是什么？
```

然后用这个框架告诉 Claude：

```bash
> 这个 bug 在业务逻辑层（CartViewModel），
> 但根本原因可能在数据层（CartRepository的缓存策略）
> 读这两个文件，用系统角度分析这个问题
```

---

## 核心思维模式 2：分层验证

**大任务分层验证，不要等到最后才发现方向错了。**

```
              /\
             /  \
            / UI  \          ← 最后验证（成本最高）
           /      \
          /业务逻辑 \         ← 中期验证
         /          \
        /  数据/算法  \       ← 最先验证（成本最低）
       /              \
      /  核心假设       \     ← 在动手之前验证
     /________________\
```

每次开始一个任务，先问：

```bash
> 我的假设是 [X]，在我们开始实现之前
> 帮我找出这个假设有没有明显的漏洞
```

---

## 核心思维模式 3：把 Claude 当"橡皮鸭" + "专家顾问"

**"橡皮鸭调试法"的增强版：**

遇到复杂问题时，先不要问 Claude 解决方案。先**向 Claude 描述问题**：

```bash
> 我来描述一个我正在思考的问题，你不用给我解决方案
> 只需要问我问题，帮我把思路理清楚
>
> 问题：我的 App 在某些设备上 launch 很慢，但我不确定瓶颈在哪里
> 开始问我问题
```

Claude 会扮演"苏格拉底式导师"，通过提问帮你自己找到答案。这种方法比直接要答案更能帮助你真正理解问题。

---

## 核心思维模式 4：情景化学习

**用具体的项目代码学习新技术，而不是看抽象教程。**

想学 Swift Concurrency？

```bash
# 不要这样
> 解释一下 Swift Concurrency 的 actor 是什么

# 要这样
> 读一下我的 DataStore.swift
> 这里有很多 DispatchQueue.main.async
> 如果我用 @MainActor 重写这些，会是什么样？
> 给我一个对比，用我自己的代码来解释 actor 的概念
```

用你自己的代码来学习，理解深度是看教程的 3 倍。

---

## 核心思维模式 5：建立反馈闭环

**每一次 Claude 表现好或者差，都是一次改进的机会。**

```
Claude 表现差了
    ↓
分析：是上下文不够？问法有问题？任务太复杂？
    ↓
调整：更精准的问法 / 更多上下文 / 任务拆解
    ↓
测试：同类问题再试一次
    ↓
总结：把好的提问方式记录在 CLAUDE.md 或个人笔记
```

经过几个月的积累，你会有一套专属于你的、高效使用 Claude Code 的方法。

## 专家的 10 个日常习惯

### 习惯 1：先读再改

任何修改之前，先让 Claude 读相关代码并总结理解，再讨论修改方案。

### 习惯 2：假设先于答案

对复杂问题，先提出假设，让 Claude 帮你验证，而不是直接要答案。

### 习惯 3：测试驱动

任何新功能，先让 Claude 生成测试描述（不是代码），确认测试覆盖了所有场景，再实现。

### 习惯 4：定期 CLAUDE.md 维护

每周花 10 分钟，把这周 Claude Code 表现不好的原因分析一下，
能写进 CLAUDE.md 的写进去。

### 习惯 5：分治复杂任务

任何任务超过 1 小时的实现，先分解成 30 分钟以内的子任务，独立完成和验证。

### 习惯 6：保持对话精简

长对话时，定期用 `/compact` 或者总结关键点开启新会话。不要让上下文无限膨胀。

### 习惯 7：给 Claude 批判性反馈

Claude 给了不好的答案时，明确告诉它哪里不好，而不是重新开始。

### 习惯 8：向 Claude 学习

Claude 给了一个你不熟悉的方案时，先理解它，再决定接受或拒绝。这是学习的机会。

### 习惯 9：记录成功的提示词

发现一个特别好用的问法？记录下来，放进你的 Prompt 库。

### 习惯 10：知道何时不用 Claude Code

简单的单文件改动、你完全熟悉的问题、快速查文档——这些不需要 Claude Code。把它用在真正值得的地方。

## 真实水平的评估

你是什么水平的 Claude Code 用户？

**初级（能用）：**
- 能让 Claude 解释代码
- 能让 Claude 修复简单 Bug
- 能让 Claude 生成样板代码

**中级（会用）：**
- 能用 CLAUDE.md 配置项目上下文
- 能管理多轮对话的上下文
- 能把复杂任务分解成步骤
- 能批判性地审查 Claude 的输出

**高级（精通）：**
- 有自定义命令和 Hooks 的工作流
- 能做多文件、跨模块的复杂任务
- Claude Code 已融入每天的工作节奏
- 能培训团队成员使用 Claude Code

**专家（1%）：**
- 用 MCP 接入外部系统
- 把 Claude Code 集成到 CI/CD
- 能在接手陌生项目 1 天内建立完整理解
- 把 Claude Code 当作思考工具，而不只是实现工具

## 最后一件事：保持人的判断

**所有工具都是工具，Claude Code 也不例外。**

它会犯错。它的知识有截止日期。它不了解你的业务。

作为开发者，你的职责是：
- **你理解你写的每一行代码**（即使是 Claude 生成的）
- **你能解释每一个架构决策**
- **你对代码质量负责**，而不是把责任推给"AI 说该这样写"

Claude Code 是让你更强大的工具，不是替代你思考的工具。

**用好它，但永远保持你的判断。**
