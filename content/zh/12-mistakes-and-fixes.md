## TL;DR

> 大多数人踩的坑都是一样的。

## 错误 1：把 Claude Code 当代码生成机器

### 症状

```
"帮我写一个用户登录功能"
→ Claude 生成了一段代码
→ 直接复制粘贴进项目
→ 发现和现有代码风格、架构完全不搭
→ 大改
→ 不如自己写
```

### 为什么会这样

Claude Code 不知道你的项目架构，生成的代码是通用的，不是为你量身定制的。

### 解法

```
# 给它上下文，让它匹配你的模式
> 读一下 Sources/Features/Profile/ 里的代码
> 我要在 Cart 模块添加类似的登录检查逻辑
> 按照 Profile 模块的模式实现，不要发明新的模式
```

**原则：让 Claude 模仿，而不是发明。**

## 错误 2：在错误的时候要代码

### 症状

```
遇到问题 → 立刻让 Claude 改代码 → 改完发现问题更多 → 还原 → 再让 Claude 改 → 循环
```

### 为什么会这样

没有先理解问题就开始动手，结果改了表象，没改根本。

### 解法

```
铁律：任何修改之前，先让 Claude 解释根因

> 分析这个问题的根本原因
> 告诉我可能的解决方案和它们的权衡
> 等我选择方案后再写代码
```

**"先诊断，后开药"——这是调试和功能开发都通用的原则。**

## 错误 3：用自然语言描述代码结构

### 症状

```
"我有一个 Manager 类，它有一些方法处理用户数据，我想把它重构成..."
→ Claude 根据你的描述生成代码
→ 和实际代码差距很大
→ 还不如直接给它看代码
```

### 解法

**不要描述代码，直接给代码。**

```bash
# 糟糕
> 我有一个 UserManager，它有 fetchUser、updateUser、deleteUser 方法...

# 正确
> 读一下 Sources/Core/UserManager.swift
> 我想把它重构成 Repository 模式
```

## 错误 4：过长的单次会话

### 症状

- 同一个 Claude 会话用了 2 个小时
- Claude 开始忘记早期讨论的决策
- 同样的问题重复出现不同答案
- Claude 的回答越来越泛泛

### 为什么会这样

上下文窗口满了，早期的信息被压缩或遗忘。

### 解法

```bash
# 每隔 30-40 分钟，做一次状态保存
> 把我们这段时间的关键决策和结论保存到 CONTEXT.md

# 开新会话时
> 读一下 CONTEXT.md，继续我们之前的工作
> 现在要做的是 [下一步]
```

## 错误 5：盲目信任 Claude 的答案

### 症状

```
Claude 说"这个问题是因为 X" → 你改了 X → 问题没解决
Claude 说"用这个 API" → 你用了 → API 已经在 iOS 17 弃用了
Claude 给了一段代码 → 你直接用 → 有一个微妙的 bug
```

### 为什么会这样

Claude 会犯错。特别是在以下情况：
- 模糊的信息输入
- 需要知道具体版本的问题
- 需要你个人偏好的判断
- 复杂的多文件依赖关系

### 解法

**对 Claude 的任何输出，都要有批判性思维。**

建立"验证习惯"：

```bash
# 验证新 API 是否有效
> 你说的这个 API 在 iOS 16 上可用吗？最低支持版本是什么？

# 验证逻辑是否正确
> 用这个测试案例走一遍你的逻辑：[具体案例]
> 结果应该是 [预期结果]，你的方案能正确处理吗？

# 验证改动的影响
> 这个改动会不会影响 [相关功能]？为什么？
```

## 错误 6：把"节省时间"当成唯一目标

### 症状

```
让 Claude 写了所有代码
→ 你不理解这些代码
→ 出问题时不知道怎么调试
→ 下次改动时破坏了之前的逻辑
→ 技术债积累
```

### 解法

**Claude Code 应该帮你更快地做你理解的事，而不是帮你跳过理解。**

当 Claude 生成了复杂的代码：

```bash
> 在你写代码之前，先解释你的实现思路
> 我要确认我理解这个方案再继续

> 写完之后，解释这段代码最关键的 3 个点
> 确保我以后能维护这段代码
```

## 错误 7：不维护 CLAUDE.md

### 症状

- 每次都要重新解释项目背景
- Claude 的代码风格和项目不一致
- 总是在纠正同样的问题

### 解法

把维护 CLAUDE.md 变成习惯：

```bash
# 发现新的项目规则时
> 把这个约定加到 CLAUDE.md：[描述约定]

# 发现错误的规则时
> 更新 CLAUDE.md，把关于 X 的规则改成 Y

# 每周五花 5 分钟
> 读一下本周我对 Claude 的纠正，看看哪些需要写进 CLAUDE.md
```

## 错误 8：一次性给太复杂的任务

### 症状

```
> 帮我把整个 App 从 UIKit 迁移到 SwiftUI，同时把架构从 MVC 改成 MVVM，
> 并且添加单元测试，顺便优化一下性能...
→ Claude 开始做，做到一半上下文满了
→ 改动混乱，无法回退
→ 项目处于半改状态
```

### 解法

**一次只做一件事。**

任何看起来"复杂"的任务，都先分解：

```bash
> 我想做 [大任务]
> 帮我把它分解成：
> - 每个步骤都是独立可提交的
> - 每步完成后项目仍然可以运行
> - 按依赖关系排序
>
> 列出所有步骤后，我们从第一步开始
```

## 错误 9：忽略 Claude 表达的不确定性

### 症状

```
Claude: "这可能是 X 问题，但我不确定，还需要更多信息来验证..."
你：直接按 X 去改
→ 改错了
```

### 解法

当 Claude 表达不确定性时，**把不确定性消除掉再行动**：

```bash
> 你说"可能"是 X 问题。帮我设计一个验证方案
> 如果是 X，会有什么可以观察到的现象？
> 如果不是 X，会有什么不同？
```

## 错误 10：用 Claude Code 做单纯的"查文档"

### 症状

```
> Swift 里的 @MainActor 是什么意思？
→ 得到一个解释
→ 其实查官方文档更准确
```

### 解法

单纯查 Swift/iOS API 文档，直接看[苹果官方文档](https://developer.apple.com)或 Dash 更准确。

Claude Code 的优势是**结合你的具体代码**来分析和应用：

```bash
# 不用 Claude 的情况
> Swift 里的 @MainActor 是什么？

# 用 Claude 更有价值的情况
> 我的这段代码（[粘贴代码]）应该用 @MainActor 还是
> DispatchQueue.main.async？分析我的具体情况给出建议
```

## 速查：常见问题诊断表

| 问题 | 最可能的原因 | 解法 |
|------|------------|------|
| Claude 给了不符合项目风格的代码 | 没有给示例/没有 CLAUDE.md | 给它相似的现有代码作为参考 |
| Claude 的分析不准确 | 上下文不够 | 给更多相关文件 |
| Claude 反复给同样的错误答案 | 问题问法有误导性 | 换个角度问 |
| 会话进行到一半效果变差 | 上下文窗口满了 | /compact 或开新会话 |
| Claude 在分析中漏掉了关键依赖 | 没有告诉它文件关联 | 明确说明文件之间的关系 |
| Claude 给了有 bug 的代码 | 边界条件描述不完整 | 在需求里明确说明边界条件 |
