# 第 8 章：构建真实项目

---

## TL;DR

> 用 Claude Code 做一个完整功能，不是"让它帮你写代码"。
> 而是让它成为你的**技术架构伙伴**——一起设计，一起实现，一起验证。
> 这章通过一个完整案例，展示正确的协作模式。

---

## 案例：为 iOS App 添加"离线模式"功能

我们将完整走一遍这个功能从 0 到完成的流程。

这个功能要求：
- 网络不可用时，展示缓存数据而不是报错
- 有离线指示器
- 恢复连接后自动同步

---

## Phase 1：需求厘清和技术调研

**千万不要上来就让 Claude 写代码。** 先做技术调研。

```bash
claude
> 我想为这个 iOS App 添加离线模式支持。
> 先读一下项目结构，特别是网络层和数据层。
>
> 然后告诉我：
> 1. 现有代码里有没有已经支持离线的基础设施？
> 2. 主要有哪些数据需要缓存（哪些 API 调用最频繁）？
> 3. 有哪些地方的代码会因为"无网络就报错"的设计而需要改？
>
> 只分析，不写代码
```

Claude 的分析会给你一个清晰的现状图，让你了解改动的影响范围。

---

## Phase 2：技术方案设计

```bash
claude
> 基于你的分析，给我 3 个实现离线模式的技术方案
> 每个方案需要说明：
> - 核心思路
> - 实现复杂度（1-5分）
> - 对现有代码的改动范围
> - 适合我们项目的原因/不适合的原因
>
> 我倾向于改动最小但效果最好的方案
```

让 Claude 给方案，你来做决策。这是正确的人机分工。

---

## Phase 3：选定方案，制定实施计划

假设选了"Cache + Network Monitor"方案：

```bash
claude
> 我选择方案2：URLCache + NetworkMonitor
>
> 帮我制定详细的实施计划：
> - 分解成独立可提交的任务（每个任务完成后代码仍然可运行）
> - 估计每个任务的复杂度
> - 标出哪些任务必须先于其他任务
>
> 用列表格式，我会把这个作为实施指南
```

**重要原则：把大功能分解成小步骤，每步都可以独立提交。**

这样的好处：
- 每步都可以测试
- 出问题容易回退
- git history 更清晰

---

## Phase 4：执行第一个任务

```bash
claude
> 开始第一个任务：创建 NetworkMonitor
>
> 要求：
> - 使用 NWPathMonitor（iOS 12+）
> - 作为单例，但支持 DI（用于测试）
> - 提供 async/await 接口和 Combine Publisher
> - 需要有单元测试
>
> 先给我代码方案（不要修改文件），等我确认后再写入
```

看完方案，提出修改意见，然后：

```bash
> 方案看起来好，有一个修改：把 singleton 改成通过 Environment 注入
> 否则测试会很麻烦
> 修改后写入文件
```

---

## Phase 5：迭代验证

每完成一个模块，立即验证：

```bash
claude
> NetworkMonitor 写完了，帮我：
> 1. 写一个简单的测试来验证它在 Simulator 上正常工作
> 2. 检查有没有内存泄漏风险（特别是 NWPathMonitor 的生命周期）
> 3. 确认 SwiftUI Preview 里还能正常运行（不要因为 Monitor 初始化崩溃）
```

---

## Phase 6：集成

```bash
claude
> NetworkMonitor 测试通过了
> 现在集成到 APIClient 里：
>
> 要求：
> - 网络不可用时，先尝试返回缓存数据
> - 缓存数据要标记来源（是实时还是缓存）
> - 如果缓存也没有，才返回网络错误
>
> 读一下当前的 APIClient.swift 实现，告诉我集成点在哪里，再动手
```

---

## Phase 7：UI 层

```bash
claude
> 现在要在 UI 层显示离线状态
>
> 需要一个 OfflineBanner：
> - 网络不可用时从屏幕顶部滑入
> - 恢复连接后自动滑出
> - 跟随 NavigationStack，不遮挡 NavigationBar
>
> 参考 Design System 里现有的 Banner 组件，保持一致的视觉风格
> 先写 SwiftUI 组件，不要集成，让我先 Preview 看看效果
```

---

## Phase 8：端到端测试

```bash
claude
> 整个离线模式功能基本完成了
> 帮我写一个 UI Test，模拟以下场景：
> 1. 正常加载数据
> 2. 断开网络连接
> 3. 验证 OfflineBanner 出现
> 4. 验证数据仍然显示（来自缓存）
> 5. 恢复连接
> 6. 验证 Banner 消失，数据刷新
>
> 使用 XCUITest + 手动控制网络（可以用 Network Link Conditioner 或者 mock）
```

---

## Phase 9：代码审查

```bash
claude
> 离线模式功能全部完成了
> 做一个整体的代码审查，检查：
> 1. 所有新代码是否符合 CLAUDE.md 里的规范
> 2. 有没有遗漏的错误处理
> 3. 有没有影响已有功能的潜在风险
> 4. 文档注释是否完整（public API）
>
> 以 Code Review 的形式输出，按优先级排列问题
```

## 这个流程的核心逻辑

```
需求分析 → 技术调研 → 方案设计 → 分解任务
    ↓
  执行 → 验证 → 集成 → 验证 → ...
    ↓
  UI → 端到端测试 → 代码审查
```

**每个阶段，你的角色：**

| 阶段 | 你做什么 | Claude 做什么 |
|------|---------|-------------|
| 需求分析 | 描述需求，回答问题 | 分析现有代码，提出疑问 |
| 方案设计 | 做决策 | 提供选项和权衡 |
| 任务分解 | 确认合理性 | 制定计划 |
| 实现 | 审查代码，提出修改 | 写代码 |
| 验证 | 运行测试，判断结果 | 写测试，分析问题 |
| 审查 | 最终决定 | 发现问题 |

**你始终是决策者，Claude 始终是执行者。**

## 常见的项目类型和特定技巧

### 接入第三方 SDK

```bash
claude
> 我需要接入 Stripe iOS SDK 做支付
> 读一下他们的文档（我给你 URL）然后：
> 1. 告诉我最简单的接入方案
> 2. 项目里有哪些需要改动的地方
> 3. 有哪些安全注意事项
```

### 从 Objective-C 迁移到 Swift

```bash
claude
> 把 LegacyManager.m / LegacyManager.h 迁移到 Swift
> 要求：
> - 接口保持向后兼容（有 OC 代码还在调用）
> - 用现代 Swift 风格（async/await 等）
> - 加单元测试覆盖主要功能
> 先分析这个类的所有公开接口，再开始迁移
```

### 添加 Widget Extension

```bash
claude
> 我要给 App 添加 iOS Widget
> 先读一下主 App 的数据层，然后：
> 1. 帮我设计 Widget 和主 App 的数据共享方案（App Group）
> 2. 创建 Widget Extension target 的基础结构
> 3. 实现一个简单的 medium size widget 显示最近订单
```

## 大型项目的节奏管理

当一个功能需要多天时：

```bash
# 每天开始时
claude
> 我们昨天完成了 NetworkMonitor 和缓存层
> 今天要做 UI 集成部分
> 快速回顾一下：当前代码状态是什么？接下来要做什么？

# 每天结束时
claude
> 今天我完成了 OfflineBanner 组件和 NavigationStack 集成
> 帮我写一个简短的进度摘要，我会保存到 PROGRESS.md
> 这样明天开始时有清晰的上下文
```
