# 第 3 章：指令的艺术

---

## TL;DR

> 你说什么，比 Claude Code 有多聪明更重要。
> 一个糟糕的问题 + 聪明的 AI = 糟糕的答案。
> 一个好问题 + 普通的 AI = 惊人的答案。
> 这章是这本书里投资回报最高的部分。

---

## 问题的本质：你在做什么

当你对 Claude Code 说话，你在做三件事：

1. **传递目标**：你想要什么结果
2. **传递上下文**：它需要知道什么背景
3. **传递约束**：什么不能做，什么必须做

大多数人只做了第一件事。

---

## 对比实验：同一个任务，两种问法

### 糟糕的问法

```
> 帮我优化这个函数的性能
```

Claude 的内心：
- 什么函数？
- 什么性能问题？内存？速度？电池？
- 优化到什么程度？
- 有什么不能改的约束？

结果：Claude 会给你一个通用的、可能完全不适用的答案。

---

### 好的问法

```
> 看一下 UserProfileViewController.swift 里的 loadUserData() 函数。
> 这个函数现在每次 viewDidAppear 都会调用，导致滑动时有明显卡顿。
>
> 我想要：
> - 加入缓存，相同 userID 10分钟内不重复请求
> - 不改变函数签名（因为有其他地方在调用）
> - 保持现有的错误处理逻辑
>
> 先读这个文件，理解现有实现，然后告诉我你的方案，等我确认再修改。
```

Claude 的内心：清晰。

结果：精准的、可直接使用的解决方案。

---

## 黄金公式：目标 + 上下文 + 约束 + 行动要求

每一个好的指令都包含这四个元素（不需要每次都全部写出，但要在脑子里想到）：

```
[目标] 我想要 X
[上下文] 背景是 Y，相关文件是 Z
[约束] 不能改动 A，必须保持 B
[行动] 先 [做什么]，然后 [等我确认/直接修改/给我建议]
```

## 六种核心指令模式

### 模式 1：探索理解型

用于：理解陌生代码、分析架构

```
> 帮我理解 [文件名/模块名]。
> 我特别想了解：[具体问题]
> 用简单的语言解释，假设我不了解这个模块的历史
```

示例：
```
> 帮我理解 NetworkManager.swift。
> 我特别想了解：请求重试逻辑是如何实现的，以及错误处理的策略
> 用简单的语言解释，假设我刚接手这个项目
```

---

### 模式 2：修复问题型

用于：修 Bug、解决 Crash

```
> 我遇到了 [具体错误信息]
> 发生在 [什么场景下]
> 相关文件是 [文件名]
> 我已经尝试过 [你试过的方法]，没有效果
> 帮我找出原因并修复
```

**关键原则：粘贴完整的错误信息，不要只说"有个报错"。**

---

### 模式 3：实现功能型

用于：添加新功能

```
> 我需要实现 [功能描述]
> 现有的相关代码在 [文件名]
> 需要符合项目的 [架构模式/编码风格]
> 先给我一个实现方案，等我确认后再写代码
```

**养成先看方案再写代码的习惯。** 避免写了一半发现方向错了。

---

### 模式 4：重构改进型

用于：提高代码质量

```
> [文件名] 里的 [函数/类] 需要重构
> 主要问题是：[问题描述]
> 重构目标：[更好维护 / 性能更好 / 更符合 Swift 惯例 / 等]
> 约束：对外接口不能变（或者说可以变）
> 先分析现有代码的问题，然后给我重构方案
```

---

### 模式 5：生成样板型

用于：快速生成重复性代码

```
> 按照 [现有示例文件] 的模式，帮我生成 [新的功能] 的样板代码
> 命名规范参考现有代码
> 需要包含：[具体内容]
```

**总是给一个现有的示例。** 让 Claude 模仿你的风格，而不是发明自己的风格。

---

### 模式 6：分析审查型

用于：Code Review、质量检查

```
> 审查 [文件名] 的代码，重点关注：
> 1. 潜在的内存泄漏
> 2. 线程安全问题
> 3. 边界条件处理
>
> 用列表形式给出问题，按严重程度排序
> 不要修改代码，只给建议
```

## 指令里的魔法词汇

这些词汇会显著改善 Claude 的行为：

| 词汇 | 效果 |
|------|------|
| "先...再..." | 控制执行顺序，避免 Claude 跳步 |
| "等我确认再修改" | 防止 Claude 直接动手改代码 |
| "只修改 X，不要动 Y" | 精确控制修改范围 |
| "给我 3 个方案" | 获得多种选择，不被第一个方案锁定 |
| "假设我是..." | 控制解释的深度 |
| "用 [语言] 回答" | 控制语言（中文/英文） |
| "简洁，不需要解释" | 让它直接给代码，不废话 |

---

## 上下文的艺术：多少是刚好

**太少**：Claude 猜测你的意图，回答偏

**太多**：上下文被无关信息占满，核心问题反而模糊

**刚好**：

```
精准 > 详尽
```

不要说"我整个项目是这样的……（1000字）"
要说"关键是这个类继承了 UIViewController 而不是 UITableViewController，
所以 dequeueReusableCell 不可用"

## 多轮对话策略

Claude Code 的多轮对话是有状态的，好好利用。

**错误方式：每次都重头说**

```
第1轮：帮我看看 UserService.swift 的问题
第2轮：帮我看看 UserService.swift 的问题，我刚才说过它的性能很差，
       现在想用缓存...
```

**正确方式：在之前的基础上递进**

```
第1轮：读一下 UserService.swift，告诉我你对这个类的理解
第2轮：好，那 fetchUser 这个方法的性能怎么样？
第3轮：我想加缓存，你有什么方案？
第4轮：用第二个方案实现，但缓存时间改成 5 分钟
```

这样的对话，Claude 的上下文是**递进积累**的，每一轮都建立在前一轮的理解上。

---

## 当 Claude 给了错误答案时

**不要：**
```
> 你说的是错的，重新来
```

**要：**
```
> 你刚才的方案有个问题：[具体说明问题是什么]
> 正确的理解应该是：[你的理解]
> 基于这个，重新给我方案
```

**修正 Claude 的方式和修正人类同事一样：具体说明问题，给出正确背景，然后让它重试。**

## 实用：iOS 开发者的常用指令模板

```bash
# 分析 Crash
> 这是 Crash 日志：[粘贴日志]
> App 版本 iOS 17，设备 iPhone 15
> 帮我找出 Crash 原因和修复方案

# SwiftUI 预览问题
> SwiftUI Preview 报错：[粘贴错误]
> 相关文件是 [文件名]
> 帮我修复，不要改变 View 的实际逻辑

# 内存泄漏
> 用 Instruments 发现 [类名] 有内存泄漏
> 相关代码在 [文件名]
> 找出泄漏位置，重点检查 closure 里的 self 引用

# 适配 iOS 版本
> [功能] 需要适配 iOS 15-17
> 当前代码只考虑了 iOS 17
> 帮我加上版本检查和向下兼容实现
```

## 本章总结：5 条原则

1. **目标 + 上下文 + 约束 + 行动**：这是好指令的四要素
2. **精准优于详尽**：不是给得越多越好，是给得越准越好
3. **先方案后代码**：复杂任务先确认方向，再动手
4. **多轮递进**：好好利用对话的连续性
5. **具体纠错**：错了就具体说哪里错了，不要光说"不对"
